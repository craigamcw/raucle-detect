# TEE-anchored VCD gate: hardware-rooted capability receipts

**Status:** Draft proposal, 2026-05-18.
**Authors:** Raucle.

## Summary

The VCD gate as specified in paper §3.3 is a software-only enforcement point whose integrity rests on standard process-isolation and key-management practices (paper §2). For regulated deployments where the gate's runtime integrity itself must be auditable — and to pre-empt likely 2027–28 EU AI Act implementing acts that mandate integrity attestation of safety controls — this proposal extends the gate with a hardware-rooted trust anchor.

The extension packages the gate as a Trusted Execution Environment (TEE) image (AWS Nitro Enclave initially, AMD SEV-SNP and Intel TDX subsequently), binds the gate's Ed25519 signing key to the TEE's measurement, and embeds a TEE attestation document into every capability receipt the gate emits. A receipt verifier can then confirm not only *what was authorised* but *that the authorising code was running the published, audited gate binary on validated hardware*.

## Threat model addition

The base threat model (paper §2) declares the gate process's integrity out of scope: an attacker who compromises the gate's runtime — RCE, key exfiltration via memory disclosure, library-load tampering — defeats every claim in the paper. This is consistent with how cryptographic gateways are usually deployed, but it leaves a gap: in deployments where the gate operator is not fully trusted by the receipt verifier (cross-organisation use cases, paper §7.5), the verifier has no evidence that the gate it claims to operate is the gate that signed the receipt.

The TEE extension closes that gap. The new assumption is that the TEE vendor's attestation chain is uncompromised (AWS Nitro's root certificate authority, AMD's KDS, Intel's Provisioning Certification Service). This is a much smaller and better-studied trust assumption than "the gate operator is honest".

## Architectural deltas

### Key provisioning

The gate's Ed25519 signing key is generated *inside* the TEE on first boot and never leaves. The public key is exported alongside the TEE's attestation document:

```
attestation = (
  measurement_pcrs,        // hash of the gate binary, build tools, kernel
  user_data,               // public_key || timestamp
  signature_from_vendor,   // Nitro root → enclave cert chain
)
```

The user-data field contains the public key the gate just generated, so any external verifier can confirm "this public key was generated by this measured binary on this validated hardware".

The corresponding private key is sealed to the TEE measurement: if the binary changes, the key is unreadable. Key rotation requires booting a fresh enclave with the new binary; the rotation document is itself a capability receipt signed by the previous enclave, naming the new public key.

### Receipt format extension

Capability receipts gain one optional field:

```jsonc
{
  // ... existing fields from paper §7.5 ...
  "teeAttestation": {
    "platform": "aws-nitro",         // or "amd-sev-snp", "intel-tdx"
    "measurement": "sha384:...",      // PCR-equivalent for the platform
    "vendorChain": "MIIE...",         // signed by AWS root / AMD KDS / Intel PCS
    "userData": "ed25519:<pubkey>||<timestamp>",
    "freshness": "nonce:..."          // anti-replay
  }
}
```

A verifier without TEE knowledge ignores the field; a TEE-aware verifier additionally checks: vendor chain valid, measurement matches the published gate binary's hash, user-data public key matches the receipt's signing key.

### Build reproducibility

The TEE measurement is only meaningful if the gate binary is reproducibly buildable. Three concrete requirements:

1. **Deterministic build.** Pinned compiler versions, hermetic dependency closure, no embedded timestamps. Achievable today via Nix or Bazel.
2. **Published measurement.** Every released gate version publishes its TEE measurement(s) — AWS Nitro PCR0/PCR1/PCR2, SEV-SNP launch measurement, TDX MRTD — alongside the binary, signed by the project's release key.
3. **Reproducible across vendors.** The same gate source produces a measurement on Nitro, a different measurement on SEV-SNP, a different measurement on TDX. Each is independently published.

A verifier consulting the published measurement registry can confirm: "the receipt's vendor-attested measurement matches the published gate version v0.13.2 on AWS Nitro".

## Why this matters

Three concrete pressures.

**1. Hyperscaler differentiation.** AWS Bedrock policy controls for agents, Google Model Armor, and Azure Prompt Shields are all software-only enforcement layers running in the hyperscaler's trust domain. None publishes a hardware-rooted attestation of the policy-enforcement code itself. A TEE-anchored gate is the only architecture in this space that produces an *operator-independent* answer to "is the gate I claim to be running actually what's running?". For cross-cloud deployments and for regulated industries that cannot fully trust their cloud vendor's internal infrastructure, this is the differentiator.

**2. EU AI Act Article 15 and implementing acts.** Article 15 requires accuracy, robustness, and cybersecurity for high-risk AI systems. The Article is in force; the implementing acts that operationalise it are in active drafting through 2026–27. Industry signal from the standardisation working groups suggests integrity attestation of safety controls is on the table. A TEE-anchored gate is the cleanest answer to that mandate, regardless of the exact wording.

**3. Cross-organisation receipts (paper §7.5).** The capability-receipt audit story collapses if the verifier has to take the operator's word for *which gate produced the receipt*. The TEE attestation closes that gap: the verifier can independently confirm that the public key in the receipt was generated by a known gate binary running on validated hardware. Without this, cross-organisation use cases reduce to "we trust that the bank's gate is the published gate" — which is the status quo and not an improvement on existing audit mechanisms.

## Non-goals

- The TEE extension does **not** prevent attacks that target the host OS, the TEE vendor's firmware, or the cryptographic primitives. The trust assumption is the TEE vendor's chain, which is a separate, well-studied risk surface.
- The TEE extension does **not** prevent prompt injection or unauthorised tool calls. Those are handled by the gate's logic, which is unchanged. The TEE extension adds *integrity attestation of the gate*, not new enforcement.
- The TEE extension does **not** mandate TEE deployment. The gate runs identically as a normal process; the attestation field is optional in receipts.

## Sequencing relative to the A2A binding

The TEE extension is independent of the A2A binding (`vcd-a2a.md`). A TEE-anchored gate can emit receipts to any consumer; an A2A-binding deployment can use a non-TEE gate. They compose cleanly:

- TEE-anchored gate emitting plain receipts → strongest in-organisation audit story.
- Software gate emitting A2A receipts → strongest cross-agent interoperability story.
- TEE-anchored gate emitting A2A receipts → both at once; the target end-state.

Strategically, the A2A binding lands first because the standards slot is time-sensitive (the binding's value collapses if Google or Microsoft write a competing profile). The TEE work lands next because its value is durable — regulatory pressure for integrity attestation will only grow.

## Open questions

1. **Which TEE platform anchors first?** AWS Nitro is the easiest (mature tooling, published PCR semantics, Python SDK). SEV-SNP and TDX are more open but harder to deploy on commodity infrastructure. Recommendation: Nitro first for reach, SEV-SNP second for openness, TDX third.
2. **How is the gate binary measurement registry distributed?** Centralised (signed publication by Raucle on a `.well-known/` URL of `raucle.com`) is simplest but creates a trust pinpoint. Distributed (sigstore-style transparency log, in-toto attestation graph) is harder but more credible for cross-organisation use.
3. **What happens on key rotation?** Booting a new enclave generates a new key. Receipts emitted by the old key remain verifiable indefinitely (the published measurement + vendor cert chain are persistent). Rotation must be signed by the outgoing enclave so a verifier can chain the new key to the old.
4. **Fallback for non-TEE-capable environments.** Some on-premise deployments (regulated banking back-office hardware, air-gapped environments) cannot run Nitro or SEV-SNP. The receipt format's TEE field is optional precisely to handle this: in those environments the gate emits non-attested receipts and the deployment's auditor falls back to standard operator-attestation procedures.
